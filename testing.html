A/B Testing

Plant Shop

Overview

I designed 2 versions of a plant shop website, and performed an A/B test on the two designs. The goal is to measure which of the two designs is better at achieving a predefined task and goal. The task for users is to checkout their cart with at least $150 of cacti/succulents. The 2 metrics that were run on statistical tests were: Time to Completion and Return Rate.

Timeline: 2 weeks

Tools: Python + Heroku Logs + Visual Studio

Context: Brown CS1300 A/B Testing Project

Context

This is one of the projects from CS1300(an UIUX class at Brown). I first designed 2 versions of the cacti website and formed statistical hypotheses. Next, I developed a website that randomly assigns version A or B to a user. Then, I sent the deployed website out to collect data. I extracted data from the Heroku logs, and transitioned the data to an excel sheet. Lastly, I cleaned and ran the data with python scripts I developed. I interpreted and analyzed the statistical results in relation to my design goals. I find this project to be particularly interesting because I learned to use data and statistical analysis to gain deeper insights into the effectiveness of my own design choices.

A test for Time to Completion

Time to Completion

I ran the T-Test from the python scripts I developed in an earlier class assignment to analyze time for a user to complete his or her $150 task. I defined the test, and explained my reasons for my alternative hypothesis below.

Definition of Time to Completion:

The time difference of when a user starts and finishes the $150 task.‚Ä®

Null Hypothesis:

The true difference between time to completion for design A and design B is zero.

Alternative Hypothesis:

The true difference between time to completion for design A and design B is different from zero.‚Ä®

A‚Äôs Layout

In design A, all items are on the left. Each row has 1 item. This will make users have to scroll down more and do more work to see all the items. Therefore, I think this layout will require more time for completion. And it‚Äôs an inefficient use of space.

B‚Äôs Layout

In design B, users don‚Äôt need to scroll down to see and add the items. This is a more optimal layout because it saves the user‚Äôs time in scrolling and completing the task. When the user first enters the page, no additional action is required to see all items.

A‚Äôs Shopping Cart Button

Shopping Cart is on the bottom left side of the page. It‚Äôs less obvious to users and they may need more time to find a shopping cart checkout button because usually it‚Äôs located at the bottom right. Also, more scrolling is required to reach the bottom of the page.

B‚Äôs Shopping Cart Button

Design B‚Äôs Shopping Cart is in a more optimal spot because the distance between the items and the button is much closer than A. Users don‚Äôt need to scroll to the bottom in order to checkout items.

Result

i failed to reject the null hypothesis. From the data output , we can see 95% of the time, the true difference will be 0. My p-value of 0.0567 is slightly larger than 0.05. So, I fail to reject the null hypothesis of no difference between my design A and design B time to completion. However I think that there is likely a difference between the 2 designs, but the sample is not big or diverse enough that we still fail to reject the null hypothesis.

CONCLUSION

What I learned about the project and A/B testing

Results

I learned that my designs for interface A and interface B were not significant for Time to Completion & Return Rate. I think the result may have been related to reasons I listed earlier such as the sample is not large and diverse enough. However, through comparing individual user sessions, I see that users mostly had a shorter time to completion and lower return rate for design B which is what I initially thought.

Take-away

In general, this is one of my favorite projects in the class because I was taught to use data to gain insights into my design. Overall, I realized a more optimal layout, navigation, and affordability can save users‚Äô time to meet their goal even though my project results were not significant. I look forward to use A/B testing in the future on my other projects.
